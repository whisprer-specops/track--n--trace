The crucial optimization: not every poll becomes a stored sample

This matters a lot.

For many metrics, you should poll often but store sparingly.

Store a historical sample only if at least one is true:

a minimum checkpoint interval elapsed

the entity is selected / watched / alerting

the value crossed a threshold

the sample is needed for export/snapshot

Otherwise:

update latest cache only

refresh last_seen

do not append redundant history

That single rule saves huge disk and memory churn.

Significance rules for your domain

Your instinct about size-based change thresholds is good, but I’d make it slightly smarter.

Use both:

absolute delta

relative delta

For example, a change is significant if:

abs(new - old) >= absolute_threshold
or

abs(new - old) / max(old, floor_value) >= relative_threshold

So for byte/volume-like metrics, something like:

absolute threshold: 10 KB

relative threshold: 10–20%

That handles both small and large values sanely.

For strings/descriptors:

store only on actual change

optionally hash + dedupe identical repeats

Concise plan

Build a local Rust engine first, not a browser-first app.

Start with 5 built-in adapter classes, all config-driven.

Use SQLite + hot cache + export snapshots, not GeoJSON as primary storage.

Keep the UI native and thin, with topology/time views as the center.

Make view influence cache priority, but not truth.

Store only significant or checkpoint samples, not every poll result.

Defer dynamic plugins until the core has been profiled under real load.

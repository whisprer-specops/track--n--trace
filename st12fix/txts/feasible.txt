Yep — and there’s one big thing to correct up front:

“1 square mile” does not meaningfully determine GeoJSON size by itself.
GeoJSON size is mostly driven by:

how many features you have

whether they are points / lines / polygons

how detailed the shapes are

how many properties each feature carries

how many decimal places you keep in coordinates

So the phrase “500 datatypes” is the part that needs decoding. That could mean at least three very different things:

500 features total

500 property fields per feature

500 distinct categories/layers/types of thing

Those lead to wildly different sizes.

Sensible rough estimates
If you mean 500 features total

For 1 square mile, approximate uncompressed GeoJSON sizes are often:

500 points with a few properties: 100 KB to 1 MB

500 lines: 500 KB to 5 MB

500 simple polygons: 1 MB to 15 MB

500 detailed polygons: 10 MB to 100+ MB

If you mean 500 properties/fields per feature

That is very heavy for GeoJSON.
Even with simple geometry, the properties alone can dominate file size.

Roughly:

1 feature with 500 short fields: maybe 10 KB to 50 KB+

500 such features: 5 MB to 25 MB+ before complex geometry

with polygons, it can climb into tens or hundreds of MB

That would generally be a sign to rethink the format or schema.

If you mean 500 different layers/categories/types

That is not impossible, but for one square mile it sounds unusually granular unless you’re doing a very rich urban/digital-twin style dataset.

A full multi-theme GeoJSON for one square mile might land around:

small / simplified: 5–20 MB

moderately rich urban data: 20–100 MB

very detailed / verbose / many attributes: 100–500 MB+

At that point plain GeoJSON starts becoming pretty clumsy.

A practical mental model

For planning purposes, use this:

Tiny GeoJSON: under 1 MB

Comfortable web-sized GeoJSON: 1–10 MB

Starting to get heavy: 10–50 MB

Probably should optimize / tile / compress: 50 MB+

Definitely wrong tool for direct delivery: 100 MB+

an IP route is not the same thing as a true geographic path.
GeoJSON can only represent this if you assign each IP / hop / inferred node some lat/lon. For real internet routing, that geographic line is at best an illustrative approximation, not a literal cable path.

Why it stays small

You only really have:

about 4–5 point features for nodes

about 3–4 line features for route segments

Even with 10 properties on each feature, that is still a tiny dataset unless you start adding:

Where size starts blowing up

The real explosion happens if by “track 10 datatypes along the route” you mean:

storing measurements at many positions along the path

repeating them over time

keeping historical snapshots

Then size becomes approximately:

snapshot size × number of time samples

So if one snapshot is 30 KB:

1,000 snapshots = ~30 MB

10,000 snapshots = ~300 MB

That is the point where plain GeoJSON starts becoming clumsy.

Yes — that sounds broadly right, and your intuition is now in a sensible range.

The main correction is this: if the snapshots are only being rendered live and not accumulated in the client as history, then you are really sizing working-set display data, not long-term storage. For 3–4 simultaneous routes, each in the rough 10–60 KB range from earlier, you are usually looking at something like 30 KB to 240 KB live, and even allowing for somewhat fatter properties or extra geometry, “a few hundred KB total at any given moment” is a very plausible estimate.

Your storage intuition is also reasonable. If one combined snapshot for all active routes were, say, 100–300 KB, then:

1 snapshot per minute for 1 hour is only about 6–18 MB

1 snapshot every 30s for 1 hour is about 12–36 MB

So yes, even if you decide to retain short-term snapshots, it stays in the MB to tens-of-MB territory rather than becoming absurdly large.

The trade-off to watch is not really raw size, but how you encode changing data. If you keep replacing one current GeoJSON object in memory, that is lightweight. If instead you append every sample into a growing FeatureCollection, size and browser overhead climb fast. Also, if the geometry does not change much between updates, it is more efficient to keep a mostly static route geometry and update only the changing metrics separately.

A cleaner design would usually be:
keep the route topology as a small GeoJSON layer, then stream the changing 10 metrics as lightweight JSON keyed by node/segment ID. That avoids re-sending near-identical coordinate data every 30–60 seconds.
